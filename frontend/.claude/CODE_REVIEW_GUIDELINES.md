# 代码评审规范

## 1. 评审目的
- 保证代码质量
- 发现潜在Bug
- 统一代码风格
- 知识分享
- 提升团队整体技术水平

## 2. 评审范围
所有提交到`develop`和`master`分支的代码都必须经过代码评审。

## 3. 评审标准
### 3.1 功能性
- 代码实现了需求描述的功能
- 没有明显的逻辑错误
- 边界情况处理正确
- 错误处理完善

### 3.2 代码质量
- 代码符合本项目的代码规范
- 变量、函数命名清晰易懂
- 没有重复代码
- 函数长度适中，不超过100行
- 注释清晰，必要的地方有说明
- 没有无用的代码和注释

### 3.3 性能
- 没有明显的性能问题
- 避免不必要的重渲染
- 接口请求合并，避免重复请求
- 大列表使用虚拟滚动

### 3.4 安全性
- 没有XSS漏洞
- 用户输入有校验
- 敏感信息不硬编码
- 接口请求有鉴权处理

### 3.5 可维护性
- 代码结构清晰
- 组件职责单一
- 耦合度低，扩展性好
- 有适当的单元测试

## 4. 评审流程
1. 开发者提交PR，填写PR模板，包含：
   - 功能描述
   - 实现思路
   - 影响范围
   - 测试情况
   - 相关Issue链接

2. 指定至少1名评审人进行评审
3. 评审人在24小时内完成评审，给出评审意见：
   - 批准：可以合并
   - 需要修改：需要根据意见修改后重新提交
   - 拒绝：严重问题，需要重新设计

4. 开发者根据评审意见修改代码，重新提交
5. 评审通过后，由开发者或负责人合并代码

## 5. 评审意见规范
- 意见要具体，指出具体的文件和行号
- 给出改进建议，而不仅仅是批评
- 态度友好，尊重开发者
- 对于有争议的问题，团队讨论决定
- 评审意见分为三类：
  - 🔴 阻塞：必须修改，否则不能合并
  - 🟡 建议：建议修改，不影响合并
  - 🟢 表扬：优秀的实现，值得学习

## 6. 评审人职责
- 认真对待每一次评审
- 对评审的代码质量负责
- 及时给出评审意见
- 帮助开发者提升代码质量
- 学习优秀的实现方式

## 7. 开发者职责
- 提交PR前进行自我检查
- 认真对待评审意见，及时修改
- 对于不认同的意见，友好沟通讨论
- 从评审中学习，提升自己的编码水平
